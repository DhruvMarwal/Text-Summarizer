Training observations
Task 1 — English → Hindi
Model trained for up to 20 epochs with early stopping (patience 15). Mixed precision (float16) enabled for faster Kaggle GPU training. Validation split of 10%. Convergence typically seen around epoch 12–16 given the truncated corpus size.
Task 3 — English → Spanish
Used 15 000 pairs from opus_books (en-es) via HuggingFace Datasets. Trained 15 epochs, no early stopping. Spanish benefited from higher lexical overlap with English, generally yielding smoother outputs vs Hindi despite fewer epochs.
Data preprocessing
English cleaned with regex (a-zA-Z only). Hindi preserved Unicode characters. Special tokens <start> and <end> added to target sequences. Sequences filtered to 2–20 word lengths to reduce noise. Post-padding applied.

Encoder–Decoder Summarizer

A FastAPI backend using facebook/bart-large-cnn paired with a clean HTML/CSS/JS frontend. Built for ATML Lab 6 as a practical demonstration of encoder–decoder sequence-to-sequence architectures.

Overview

This project has two parts. The backend (summarizer_backend.py) is a FastAPI server that loads BART and exposes a POST /summarize endpoint. The frontend (distill.html) is this file — a standalone HTML page that calls that endpoint and visualises results.

BART (Bidirectional and Auto-Regressive Transformer) is an encoder–decoder model pre-trained with a denoising objective, then fine-tuned on CNN/DailyMail for summarisation. This makes it well-suited to abstractive summarisation where the output is fluent, human-readable prose rather than extracted sentences.

Setup & installation

Install Python dependencies:

pip install fastapi uvicorn transformers torch sentencepiece

Start the backend server:

uvicorn summarizer_backend:app --reload --port 8000

Open the frontend — just double-click distill.html in your file browser, or serve it locally:

python -m http.server 3000
# then visit http://localhost:3000/distill.html
First run: BART (~1.6 GB) is downloaded from HuggingFace on first startup. This can take a few minutes depending on your connection. Subsequent runs use the local cache.

Using the summarizer

  • Paste any text into the Input panel (or click "load demo" for a sample).
  • Adjust Max and Min length sliders to control summary verbosity.
  • Click Summarize — the pipeline animation shows inference progress.
  • Use the copy button to grab the summary. The compression metric shows how much shorter the output is.
  • Texts over ~1 000 tokens are automatically chunked; partial summaries are merged and re-summarised.

Architecture explained

Encoder (bidirectional transformer)
Takes the input token sequence and produces contextual representations for every position. Self-attention allows each token to attend to every other token in both directions simultaneously — unlike LSTM which processes left-to-right.
Cross-attention bridge
At each decoder step, cross-attention queries the full encoder output to decide which parts of the input are most relevant for generating the next token. This is what allows abstractive behaviour — the decoder can rephrase and restructure content freely.
Decoder (auto-regressive)
Generates output tokens one at a time, left to right. Each new token conditions on all previously generated tokens (masked self-attention) plus the encoder's contextual representations (cross-attention). Beam search (n=4) is used at inference for better quality.

API reference

GET /health — returns {"status":"ok"}. Use to check if the server is running.

GET / — returns server status and active model name.

POST /summarize

{
  "text":       "string  (required)",
  "max_length": 130,     // int, 30–500, default 130
  "min_length": 30       // int, 10–200, default 30
}

Response:

{
  "summary":      "string",
  "input_tokens": 312,
  "output_tokens": 87,
  "model":        "facebook/bart-large-cnn"
}

If the BART model fails to load (e.g. no internet, no GPU), the backend falls back to a local extractive frequency-based summariser automatically. The model field in the response will say extractive-frequency-fallback in that case.

Lab context

This frontend was built for ATML Lab 6, which focused on encoder–decoder architectures. The lab consisted of three tasks:

  • Task 1: Build an LSTM-based seq2seq model for English–Hindi translation using the Hindi–English Truncated Corpus.
  • Task 2: Evaluate the Task 1 model using BLEU score (smoothed, n=30 samples).
  • Task 3: Train a new seq2seq model for English–Spanish translation using 15 000 pairs from the opus_books dataset via HuggingFace.

The summarizer backend uses a pre-trained BART model rather than a trained-from-scratch model, demonstrating how the same encoder–decoder principle scales from small LSTM experiments to large transformer architectures.
